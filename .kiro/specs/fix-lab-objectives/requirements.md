# Requirements Document

## Introduction

This document specifies the requirements for fixing the Smart IT Lab labs system where objectives remain NOT COMPLETED even after executing all Useful Commands. The system currently has a mismatch between the commands shown in the frontend, the command outputs generated by terminal-engine.js, and the objective validation logic in lab.controller.js. This fix will ensure that when users click and execute all Useful Commands in order, they achieve 100% completion for ALL labs.

## Glossary

- **Lab**: A learning exercise with objectives, topology, and commands defined in seed.js
- **Objective**: A specific learning goal within a lab that must be completed
- **Useful_Command**: A command displayed in the frontend left panel that users can click to execute
- **Terminal_Engine**: The backend module (terminal-engine.js) that generates simulated Cisco IOS output for commands
- **Objective_Validator**: The logic in lab.controller.js that checks if a command satisfies an objective
- **ObjectiveCommands**: An array in the lab definition mapping each objective to trigger keywords
- **Score**: Percentage calculated as (completed objectives / total objectives) × 100
- **Lab_Instance**: A UserLab record tracking a user's progress through a specific lab

## Requirements

### Requirement 1: Lab Definition Audit

**User Story:** As a developer, I want to audit all lab definitions, so that I can identify which labs have mismatched Useful Commands and objectives.

#### Acceptance Criteria

1. WHEN reading the seed file, THE System SHALL extract all labs with their labId, name, objectives array, commands array, and objectiveCommands array
2. WHEN a lab has N objectives, THE System SHALL verify that the commands array contains at least N entries
3. WHEN a lab has N objectives, THE System SHALL verify that the objectiveCommands array contains exactly N entries
4. WHEN objectiveCommands[i] is missing or empty, THE System SHALL flag objective i as unmapped
5. WHEN all labs are audited, THE System SHALL produce a report listing labs with mapping issues

### Requirement 2: Terminal Engine Command Coverage

**User Story:** As a developer, I want to verify that all Useful Commands produce output in terminal-engine.js, so that the terminal displays realistic responses.

#### Acceptance Criteria

1. WHEN a Useful Command is executed, THE Terminal_Engine SHALL return output that is not empty
2. WHEN a Useful Command matches a pattern in OUTPUTS map, THE Terminal_Engine SHALL return the corresponding Cisco IOS output
3. WHEN a Useful Command does not match any pattern, THE Terminal_Engine SHALL return a realistic "command not found" or "invalid syntax" error
4. WHEN terminal output is generated, THE Terminal_Engine SHALL include keywords that match the objective validation patterns
5. WHEN all Useful Commands for a lab are tested, THE Terminal_Engine SHALL produce output for 100% of them

### Requirement 3: Objective Validation Pattern Matching

**User Story:** As a developer, I want to ensure objective validation patterns match the terminal output, so that objectives are marked complete when the correct command is executed.

#### Acceptance Criteria

1. WHEN a command is executed, THE Objective_Validator SHALL normalize the command by converting to lowercase and trimming whitespace
2. WHEN checking objective i, THE Objective_Validator SHALL retrieve objectiveCommands[i] from the lab definition
3. WHEN objectiveCommands[i] contains multiple trigger keywords, THE Objective_Validator SHALL mark the objective complete if ANY keyword is found in the normalized command
4. WHEN a command matches an objective trigger, THE Objective_Validator SHALL add the objective index to completedObjectives array
5. WHEN multiple objectives match a single command, THE Objective_Validator SHALL mark ALL matching objectives as complete

### Requirement 4: Score Calculation Accuracy

**User Story:** As a student, I want the score to accurately reflect my progress, so that I know when I have completed all objectives.

#### Acceptance Criteria

1. WHEN calculating score, THE System SHALL use the formula: (completedObjectives.length / totalObjectives) × 100
2. WHEN 4 out of 5 objectives are completed, THE System SHALL display score as 80%
3. WHEN all objectives are completed, THE System SHALL display score as 100%
4. WHEN score reaches 100%, THE System SHALL set lab status to "completed"
5. WHEN score is below 100%, THE System SHALL NOT set lab status to "completed"

### Requirement 5: Useful Command Click-to-Execute

**User Story:** As a student, I want Useful Commands to execute automatically when clicked, so that I can complete labs efficiently.

#### Acceptance Criteria

1. WHEN a user clicks a Useful Command button, THE System SHALL populate the terminal input field with the command text
2. WHEN the command text is populated, THE System SHALL automatically submit the command to the backend
3. WHEN the command is submitted, THE System SHALL display the output in the terminal
4. WHEN the command completes, THE System SHALL clear the input field
5. WHEN the command triggers objective completion, THE System SHALL update the objectives panel immediately

### Requirement 6: Lab Definition Repair

**User Story:** As a developer, I want to repair all lab definitions with missing or incorrect objectiveCommands, so that all labs can be completed.

#### Acceptance Criteria

1. WHEN a lab has missing objectiveCommands entries, THE System SHALL generate appropriate trigger keywords based on the commands array
2. WHEN generating trigger keywords, THE System SHALL use command prefixes that uniquely identify the command (e.g., "show ip ospf" for "show ip ospf neighbor")
3. WHEN multiple commands can satisfy an objective, THE System SHALL include all valid command prefixes in objectiveCommands[i]
4. WHEN objectiveCommands are updated, THE System SHALL preserve the original objectives array unchanged
5. WHEN all repairs are complete, THE System SHALL update seed.js with the corrected lab definitions

### Requirement 7: Terminal Engine Output Repair

**User Story:** As a developer, I want to add missing command outputs to terminal-engine.js, so that all Useful Commands produce realistic responses.

#### Acceptance Criteria

1. WHEN a Useful Command has no matching pattern in OUTPUTS, THE System SHALL add a new entry to the OUTPUTS map
2. WHEN generating output, THE System SHALL use realistic Cisco IOS format with proper headers, columns, and data
3. WHEN output is generated for a show command, THE System SHALL include the keywords that the objective validator expects
4. WHEN output is generated for a configuration command, THE System SHALL return an empty string or confirmation message
5. WHEN all missing outputs are added, THE System SHALL update terminal-engine.js with the new OUTPUTS entries

### Requirement 8: End-to-End Validation

**User Story:** As a QA engineer, I want to validate that all labs can reach 100% completion, so that students will not encounter incomplete labs.

#### Acceptance Criteria

1. WHEN testing a lab, THE System SHALL execute all Useful Commands in the order they appear
2. WHEN all commands are executed, THE System SHALL verify that completedObjectives.length equals objectives.length
3. WHEN all objectives are complete, THE System SHALL verify that score equals 100
4. WHEN score is 100, THE System SHALL verify that lab status is "completed"
5. WHEN validation fails for any lab, THE System SHALL report the lab name, failing objective index, and expected vs actual state

### Requirement 9: Objective Validation Logic Correction

**User Story:** As a developer, I want to fix objective validation patterns that incorrectly reject valid command output, so that objectives complete when they should.

#### Acceptance Criteria

1. WHEN an objective expects "OSPF routes in routing table", THE Objective_Validator SHALL accept commands containing "show ip route ospf"
2. WHEN terminal output contains the expected keywords, THE Objective_Validator SHALL mark the objective complete regardless of output formatting
3. WHEN validation logic uses pattern matching, THE System SHALL use case-insensitive substring matching
4. WHEN validation logic is updated, THE System SHALL preserve backward compatibility with existing completed labs
5. WHEN validation patterns are corrected, THE System SHALL update lab.controller.js with the improved logic

### Requirement 10: Comprehensive Lab Coverage

**User Story:** As a developer, I want to fix ALL labs in the system, so that no lab is left with incomplete objectives.

#### Acceptance Criteria

1. THE System SHALL process all labs in seed.js including original labs (lab-1 through lab-4) and module labs (lab-m1-1 through lab-m7-2)
2. THE System SHALL NOT skip any lab regardless of difficulty level (beginner, intermediate, advanced)
3. THE System SHALL NOT modify lab names, descriptions, or objectives text
4. THE System SHALL NOT change the scoring formula or completion criteria
5. WHEN all labs are processed, THE System SHALL verify that 100% of labs can reach 100% completion through their Useful Commands
